## 先确定读者问题

生成图表前依次回答：

1. 读者是谁：产品、研发、运维、安全、外部集成方还是管理者？
2. 读者要回答什么问题：边界、依赖、调用顺序、状态、部署、数据归属还是交付时间？
3. 图的范围是什么：企业、系统、部署单元、模块、类、单次请求还是单个对象生命周期？
4. 哪一种关系最重要：包含、依赖、消息、控制流、数据流、状态迁移或时间？
5. 哪些细节不属于这张图？把它们移到下一层视图、表格或正文。

一张图只回答一个主要问题。标题应直接写明该问题，例如“Server 模块依赖方向”“设备首次绑定时序”“生产部署拓扑”，不要只写“架构图”或“流程图”。

## 架构与 UML 视图选择

| 读者问题 | 首选视图 | 关键元素 | 不要使用 |
| --- | --- | --- | --- |
| 系统服务谁、依赖哪些外部系统 | C4 System Context | 人、目标系统、外部系统、关系 | 内部类和接口清单 |
| 系统由哪些可部署/可运行单元组成 | C4 Container | 应用、服务、数据库、消息系统、协议 | 把源码目录当 Container |
| 一个部署单元内部有哪些职责模块 | C4 Component / UML Component | 模块、公开接口、单向依赖、边界 | 带十几个步骤编号的流程线 |
| 代码包如何分层、谁可以依赖谁 | UML Package / 模块依赖图 | Package、Group、允许方向、禁止方向 | 业务操作流程图 |
| 一次请求由谁按什么顺序完成 | UML Sequence / `sequenceDiagram` | 参与者、生命线、按时间排列的消息 | 横向平铺消息节点 |
| 一个对象有哪些状态及触发条件 | UML State Machine / `stateDiagram-v2` | 状态、事件、守卫、终态 | 把状态写成普通步骤 |
| 类型、接口、继承与组合是什么 | UML Class / `classDiagram` | 类、成员、继承、关联、多重性 | 展示运行时调用顺序 |
| 数据实体、字段和基数是什么 | ER / `erDiagram` | 实体、字段、主外键、基数 | 用箭头暗示不明确的数据关系 |
| 数据从哪里产生、转换和落地 | Data Flow / Pipeline | Source、Transform、Store、Sink | 同时混入部署和调用时序 |
| 业务步骤、条件和回退是什么 | Activity / `flowchart` | 动作、判断、分支、合并、终点 | 静态模块依赖 |
| 软件部署到哪些节点和网络区 | C4 Deployment / UML Deployment | 环境、节点、实例、网络边界 | 用源码模块代替运行实例 |
| 多角色之间如何交接工作 | Swimlane / 分组流程 | 责任泳道、步骤、交接 | 只有角色、没有顺序的关系图 |

UML 是语义工具箱，不是必须一次用齐的模板。C4 官方也明确不要求绘制全部层级；多数团队从 System Context 与 Container 开始，只在确有价值时增加 Component、Dynamic 或 Deployment 视图。

## Docs Engine 表达映射

| 目标视图 | 当前首选输入 | Board 语义 |
| --- | --- | --- |
| 业务流程 / Activity | Mermaid `flowchart` | 节点、条件边和方向 |
| 跨角色协议 / Dynamic | Mermaid `sequenceDiagram` | 参与者、生命线、纵向消息顺序 |
| 状态机 | Mermaid `stateDiagram-v2` | 状态节点和触发边 |
| 类和接口 | Mermaid `classDiagram` | 类型节点、成员和关系 |
| 数据模型 | Mermaid `erDiagram` | 实体节点、字段和基数 |
| 模块 / Package / Component | Mermaid `flowchart` + `subgraph`，或直接写 `BoardDocument` | Group/Container、成员节点、跨组依赖 |
| 精确设计的架构板 | 直接写 `BoardDocument` 与 authored layout | 作者位置、Group、锚点和路线 |

Mermaid 只是输入格式。不要依赖 Mermaid 运行时主题、SVG 输出或未验证的完整语法；输入会被导入成统一 BoardDocument，再由 Docs Engine 渲染和编辑。

静态架构关系使用 Group/Container 表达边界。不要为了得到一个外框把模块伪装成流程步骤，也不要用颜色代替分组名称。

## 复杂度与拆图门槛

以下是默认写作门槛，不是运行时强制限制。触发任一项时，先拆图或提升抽象层级：

- 同一张图表达两种以上主要关系，例如同时表达模块依赖、运行时消息和部署位置。
- 节点超过 12 个、关系超过 15 条，且读者必须逐条追踪。
- 跨 Group 连线超过 3 条，或同一对 Group 之间出现多条方向不同的连线。
- 一条边标签需要两行以上才能读完，或边标签承载请求体、Token、字段清单等实现细节。
- 存在 8 个以上编号消息，仍试图使用 flowchart 或模块依赖图。
- 自动布局出现交叉线、回头线或共享线段，读者无法从起点连续追到终点。
- 为解释一张图需要连续写两段以上“某种线表示什么”。

拆图时保留稳定导航：先给高层边界图，再分别链接依赖图、关键时序图、状态图或部署图。不要简单把一张大图按屏幕区域切成多张，必须让每张图拥有独立问题。

## 节点、分组和连线规则

- 节点使用读者语言和稳定职责名称，不使用无意义内部 ID。
- Group 名称表示真实边界：部署单元、模块域、责任方、网络区或生命周期阶段。
- Group 内只放属于该边界的对象；跨组对象不能为了减少连线而重复出现。
- 静态依赖边使用“调用/读取/发布/实现”等关系语义；时序消息使用具体动作或接口。
- 线型最多表达一种额外维度，并在正文解释。例如实线表示业务依赖、虚线表示装配；不要再同时让颜色、粗细和箭头形状编码三套规则。
- 不合并会丢失来源或目标身份的多条边。共享主干只能用于明确的总线、Topic 或同一逻辑通道。
- 时序图按时间向下；参与者按职责排列。请求与响应成对，长载荷移到正文或表格。
- 模块图保持依赖方向一致。出现循环时直接标记架构债务，不通过折线路由掩盖。

## 架构文档的推荐组合

一个完整架构解释页面通常从以下视图中选择 2–4 张，而不是画一张万能流程图：

1. System Context：产品、用户与外部系统边界。
2. Container 或部署单元图：进程、应用、数据库与协议。
3. Component / Package 图：目标模块内部的职责和允许依赖。
4. Sequence：一条最关键的运行时链路。
5. State：存在复杂生命周期时补充。
6. Deployment：生产环境、网络边界或容量是阅读重点时补充。

例如账号与设备身份文档应把“身份组件边界”与“首次绑定时序”分开；12 条编号消息属于 Sequence，不应压进横向组件图。模块化单体文档应把“模块 Owner/边界”与“一轮语音调用链”分开；组合根注入、业务依赖和遥测如果造成蜘蛛网，应拆成不同视图。

## 发布前检查

- [ ] 图表类型直接回答本节标题提出的问题。
- [ ] 每张图只有一个主要关系和一个抽象层级。
- [ ] 架构边界使用 Group/Container，而不是只靠节点颜色。
- [ ] 编号消息使用 Sequence，状态使用 State，字段基数使用 ER。
- [ ] 跨组边、循环和架构债务没有被自动布局隐藏。
- [ ] 边标签简短；接口载荷和字段细节已移到正文、表格或代码块。
- [ ] 正文给出读图结论，而不是逐像素描述图形。
- [ ] 构建后检查嵌入和全屏模式，确认标签、箭头、容器标题和生命线完整。
