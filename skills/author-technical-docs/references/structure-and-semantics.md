# 结构与语义决策

## 快速决策表

| 内容关系 | 首选结构 | 适用条件 | 不要用于 |
| --- | --- | --- | --- |
| 连续论述 | 段落 | 一个结论及其原因、证据或边界 | 人工做成卡片墙 |
| 有顺序的动作 | 有序列表 | 步骤、阶段、优先级 | 无顺序的并列事实 |
| 无顺序的集合 | 无序列表 | 选项、条件、示例、检查项 | 只有一个项目 |
| 重复字段的多对象比较 | 表格 | 每一行共享相同列定义 | 布局、长篇叙述、单列步骤 |
| 数学或概念关系 | `Formula` | 符号结构能显著加快理解 | 配置赋值、代码、口号、普通引用 |
| 标识符或用户输入 | 行内代码 | 类型、字段、路径、命令、短值 | 产品名、普通术语、强调 |
| 可执行或可复制片段 | 代码块 | 多行代码、命令、配置、输出 | 普通说明或数学公式 |
| 来源原话 | `blockquote` | 必须保留原始措辞并可说明来源 | 公式、提示、总结、装饰卡片 |
| 页面结论 | `SummaryPanel` | 页面级摘要或决策 | 每节重复使用 |
| 关键判断 | `Annotation` | 需要在正文中突出的一条结论 | 充当标题或目录层级 |
| 风险、限制或行为提示 | `Callout` | 读者必须据此改变操作或判断 | 普通背景段落 |
| 状态或优先级 | `Status` / `Priority` | 值来自稳定枚举 | 代替完整结论 |
| 前后方案对照 | `Transition` | 明确的起点、目标和变化理由 | 多维精确比较 |
| 多项验收或风险 | `RiskGrid` | 每项均可独立判断 | 普通工作进展 |
| 有日期范围的计划 | `Timeline` | 阶段、依赖、起止时间重要 | 单个日期或纯历史叙述 |
| 实体关系或事件流 | 原生图表 | 关系本身比单个实体更重要 | 两个节点即可说清的简单关系 |

## 公式判断

按顺序回答：

1. 内容是否是程序语法、赋值、配置或 API 示例？是则使用代码。
2. 内容是否只是自然语言中偶然包含 `=`、`×`、`+`？是则保留正文。
3. 符号结构是否表达稳定的数学、度量或概念模型，并且独立排版能降低理解成本？是则使用显式 `Formula`。
4. 关系是否需要解释变量、范围、单位或假设？在公式前后用正文解释，不把说明全部塞进 `\text{}`。

推荐：

```mdx
<Formula
  aria-label="转化率等于完成目标行为的用户数除以进入流程的用户数"
  latex={String.raw`\text{转化率} = \frac{\text{完成目标行为的用户数}}{\text{进入流程的用户数}}`}
/>
```

不推荐：

```mdx
> 转化率 = 完成用户数 ÷ 进入用户数
```

不要依赖 blockquote、正则表达式或渲染时文本扫描把后者转换为公式。此类推断会误伤引用、代码、全角标点和本地化内容。

## 图表类型

下表只用于快速选择基础语法。涉及软件架构、UML、模块依赖、部署拓扑、数据流或复杂图表拆分时，继续读取 [图表选择与编排](diagram-selection.md)。

| 关系问题 | 图表类型 | 典型场景 |
| --- | --- | --- |
| 下一步是什么，条件如何分支 | `flowchart` | 业务流程、审批、故障排查、决策门槛 |
| 谁在什么时候向谁发送什么 | `sequenceDiagram` | API 调用、设备与服务交互、登录流程 |
| 对象会处于哪些状态并如何迁移 | `stateDiagram-v2` | 生命周期、播放状态、订单状态 |
| 类型、继承和成员是什么 | `classDiagram` | 代码结构、领域模型 |
| 实体、字段和基数是什么 | `erDiagram` | 数据模型、数据库关系 |
| 阶段何时开始和结束 | `gantt` 或 `Timeline` | 交付计划、研究阶段、项目排期 |
| 分支、提交和合并如何演进 | `gitGraph` | 发布分支、补丁回合、版本历史 |
| 事件按时间如何发生 | `timeline` | 发布历史、事故过程、演进路线 |
| 概念如何分层归类 | `mindmap` | 主题拆解、知识分类 |
| 少量类别如何构成整体 | `pie` | 只强调占比且类别很少 |

使用 Docs Engine 当前支持的语法子集，不要把上游 Mermaid 的全部语法当作兼容承诺。创建或修改图表后必须构建并检查：

- 节点标签不得退化为内部 ID。
- 箭头方向、边标签、循环和条件必须与正文一致。
- 颜色不能是唯一的状态或分类信号。
- 图表必须有描述主题或用途的可访问名称。
- 精确数值比较优先用表格；不要用饼图隐藏微小差异。

## 表格与列表

- 同一对象只有一个属性时使用列表；多个对象共享多个属性时使用表格。
- 表头必须描述数据含义，不能只靠列位置或颜色传达。
- 单元格保持简短；需要多段解释时改为小节或链接到详情。
- 数字采用一致单位和精度，并在表头或正文说明口径。
- 不用表格控制页面布局。

## 强调组件

- `SummaryPanel`：一页通常最多一个，先给读者页面级结论。
- `Annotation`：突出一句可判断的结论，不生成伪标题。
- `Callout`：按风险、限制、信息、成功等真实语义选择变体；不要按喜好选颜色。
- `Transition`：只用于明确的“当前 → 目标”变化，分别说明起点问题、目标状态和变化逻辑。
- `RiskGrid`：每项必须可判断或可验收，避免把普通描述拆成装饰卡片。
