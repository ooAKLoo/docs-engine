# 专业技术文档创作

## 目标

把原始材料组织成读者可以快速理解、准确执行、长期维护的文档。先确定信息语义，再选择 Markdown、MDX 或 Docs Engine 组件；不要为了视觉效果滥用结构。

## 核心边界

- 在文档源文件中显式表达语义。渲染器只负责展示，不根据标点、关键词或文字形状猜测内容类型。
- 不在依赖项目复制样式、包装渲染器或添加兼容补丁。优先使用 Docs Engine 已提供的语义组件；缺少能力时记录引擎缺口。
- 不把 `blockquote`、标题、表格、代码块或图片当作通用卡片。每种结构只承担其真实语义。
- 不以“更丰富”为目标堆叠卡片、颜色和图表。没有改善理解速度或准确性的视觉结构应删除。
- 保留用户事实、术语、约束与不确定性。没有证据时不得补造数据、结论、接口或执行结果。

## 工作流

### 1. 检查项目上下文

1. 阅读项目中的 `AGENTS.md`、内容规范、已有同类文档和构建命令。
2. 确认目标文件类型、Docs Engine 版本、可用组件及宿主支持的语法。
3. 识别读者、阅读场景、先验知识、要完成的任务和验收结果。
4. 区分用户提供的事实、可验证事实、推断与待确认项。

不要仅凭另一项目或旧版本的组件名写 MDX。先从当前依赖或导出入口确认组件存在。

### 2. 确定页面主类型

为每个页面选择一个主要目的：

- 教程：让初学者在安全引导下完成一次学习体验。
- 操作指南：帮助已有基础的读者解决一个真实任务。
- 参考手册：准确、完整、中性地描述接口、字段、配置或约束。
- 解释文章：说明背景、原理、权衡和“为什么”。

不要把四种目的平均混在同一条叙事中。确实需要混合时，以一个主类型组织正文，把其他内容放入短提示或独立页面并互相链接。

### 3. 设计信息层级

1. 用唯一 H1 概括页面任务或主题。
2. 在开头说明对象、目的、适用范围和读完可获得的结果。
3. 按读者决策或执行顺序组织 H2；只在 H2 下使用 H3，不跳级。
4. 把前置条件放在操作之前，把结果验证放在操作之后。
5. 把背景解释、完整字段表和边界情况移到读者真正需要的位置。
6. 使用能独立传达主题或动作的标题，避免“其他”“更多”“说明”等空泛标题。

### 4. 选择语义结构

遇到公式、对比、状态、流程、时序、风险或高亮信息时，读取 [结构与语义决策](references/structure-and-semantics.md)。严格按内容关系选择结构，而不是按期望颜色或外形选择组件。

特别遵守以下规则：

- 数学或概念关系需要独立阅读时，显式使用 `<Formula>` 和 LaTeX。
- 赋值、配置、命令、类型、路径和 API 字段使用行内代码或代码块，不使用公式。
- `>` 只用于来源原话或必须保留原始措辞的引用。
- 顺序重要的操作使用有序列表；无顺序集合使用无序列表；存在多个重复字段时使用表格。
- 三个以上实体存在依赖、分支、循环、状态迁移或跨角色交互时，考虑原生图表。
- Callout 只承载需要改变读者行为的提示、风险、限制或结果，不承载普通段落。

### 5. 显式编写公式

Agent 必须在写作阶段判断公式语义，不得要求运行时自动识别“结果 = 因子 × 因子”。使用语义清晰的 LaTeX：

```mdx
<Formula
  aria-label="用户体验等于跨渠道连续性乘以上下文相关性乘以记忆可靠度乘以用户控制权"
  latex={String.raw`\text{用户体验} = \text{跨渠道连续性} \times \text{上下文相关性} \times \text{记忆可靠度} \times \text{用户控制权}`}
/>
```

- 中文概念放入 `\text{}`。
- 乘法、除法、约等于和不等式分别使用 `\times`、`\div`、`\approx`、`\le`、`\ge` 等命令。
- 为领域公式提供可读的 `aria-label`；不要把 LaTeX 源码当作无障碍说明。
- 简短且不影响断行的关系可保留在正文；复杂、核心或多项关系使用独立公式块。
- 不把口号、普通并列句、配置赋值或代码表达式包装成公式。
- 不在内容文件写公式背景色、字体或 KaTeX HTML。视觉与安全渲染由 Docs Engine 维护。

### 6. 编写与校对

- 每段先给结论或动作，再给原因、证据和边界。
- 使用一致术语；首次出现时解释缩写，后文不要换同义叫法。
- 用主动句和具体动词，删除“进行相关处理”“能够实现”等空泛表达。
- 每个列表项保持相同语法结构；每个步骤只包含一个主要动作和可观察结果。
- 代码示例必须最小、可复制、标注语言，并明确省略内容和预期输出。
- 链接文本描述目标，不使用“点击这里”；不要裸露冗长 URL 代替名称。
- 图表节点、边标签和表头使用读者语言，不暴露无意义内部 ID。

### 7. 验证成品

完成后读取 [质量检查清单](references/quality-checklist.md) 并执行与项目匹配的检查：

1. 运行文档构建、类型检查、链接检查或项目规定的验证命令。
2. 检查桌面与窄屏下的标题层级、表格滚动、代码复制、公式溢出和图表标签。
3. 检查浅色与深色主题、键盘访问、替代文本以及颜色之外的语义提示。
4. 确认引用仍是引用、公式均为显式组件，且依赖项目没有新增渲染兼容层。
5. 如果无法渲染验证，明确说明未验证的范围，不宣称视觉结果已通过。

## 输出要求

- 优先直接修改用户指定的文档；不要无故扩展到运行时组件或宿主配置。
- 交付时简述页面类型、关键结构决策、验证结果及仍需确认的事实。
- 当内容问题源于 Docs Engine 能力缺口时，把问题定位到引擎并提出最小接口需求；不要让每个依赖项目分别修补。
